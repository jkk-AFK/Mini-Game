# Stage 4 · Approve

## 1. 执行检查清单（用于每个任务开始前）

- [ ] 明确该任务的需求、输入输出、验收标准。
- [ ] 确认依赖任务均已完成并经审查通过。
- [ ] 更新任务看板状态并登记负责人及开始时间。
- [ ] 准备所需的开发环境、样例数据、测试脚本。
- [ ] 评估潜在风险（性能、安全、兼容性）并记录应对方案。
- [ ] 通知相关协作者（如实时服务或游戏模块的接口人）任务即将开始。

## 2. 迭代执行检查清单（开发过程中循环执行）

- [ ] 需求变更或疑问及时记录，并更新文档及任务描述。
- [ ] 编写/更新单元测试与端到端测试，确保覆盖关键路径。
- [ ] 代码通过本地 lint、format、type-check、unit-test。
- [ ] 中间成果及时推送至远程仓库/CI 进行验证。
- [ ] 对实时依赖（WebSocket/数据库）进行联调验证。
- [ ] 记录本次迭代中的问题、解决方案与遗留风险。

## 3. 完成确认清单（任务提交前）

- [ ] 代码变更经自审并完成同行评审。
- [ ] 所有测试通过并附带测试报告或截图。
- [ ] 文档（API、架构、操作指南）同步更新。
- [ ] 部署脚本/配置更新并在开发环境验证。
- [ ] 安全审查：依赖检查、密钥管理、输入验证。
- [ ] 更新任务状态为完成，附上链接（PR/文档/测试报告）。

## 4. 阶段质量门控

- [ ] 所有任务均有对应的验收记录与可追溯文档。
- [ ] 关键任务（T4-T7、T11-T15）需额外安全与性能验证。
- [ ] 全局回归测试覆盖注册登录、游戏启动、成绩上报、排行榜、管理员操作。
- [ ] 国际化、无障碍、响应式适配通过人工和自动化抽检。
- [ ] 实时通信在最少 4 名玩家并发条件下测试延迟与稳定性。
- [ ] 若出现重要缺陷，回溯至受影响阶段重新执行并记录。
